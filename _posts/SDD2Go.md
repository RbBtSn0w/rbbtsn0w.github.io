这是个非常值得深思的战略问题。让我从几个维度来分析。信息非常充分，可以给出一个有深度、有立场的判断了。

---

## 一、先直接回答核心问题

**LLM 会在内部"解决掉"SDD 的问题吗？**

**部分会，但不会全部。** 而且更重要的是：LLM 能力提升和 SDD 方法论的演进，不是替代关系，而是共同进化关系——模型越强，对"意图表达"的要求反而越高，不是越低。

---

## 二、LLM 技术演进能解决的部分

**上下文腐烂（Context Decay）——正在被解决**

跨多个上下文窗口让 agent 保持一致进展，是目前尚未解决的核心问题。每个新会话在没有任何前置记忆的情况下启动，就像一个工程团队轮班制工作，每个新工程师来了却完全不记得上一班发生了什么。

Anthropic 的工程实践给出的临时解法仍然依赖文件系统——开发了一个两阶段解法：初始化 agent 在第一次运行时建立环境，编码 agent 在每次会话中增量推进，并在会话结束时保持"干净状态"——没有明显 bug、代码有序且有良好文档，下一个开发者可以轻松接手。 这本质上还是 SDD 的逻辑，只是在 agent 层自动化了。

研究人员指出"上下文腐烂"（context rot）现象——单纯扩大上下文窗口导致性能下降，缺少上下文管理会使 AI agent 的响应不准确或不可靠。 这说明无限上下文不是银弹。

学术界的最新研究把 agent 记忆分为三种：事实记忆（factual）、经验记忆（experiential）、工作记忆（working），并分析记忆如何随时间形成、演化、被检索。 这个方向正在快速发展，2026-2027 年内技术成熟度会显著提升。

**结论：** 上下文持久化这个问题，技术层面 1-2 年内会有重大突破，但不会"消失"——它会从"开发者手动管理文件"变成"agent 自动管理记忆层"，SDD 的 spec 文件会成为这个记忆系统的输入源，而不是被取代。

---

## 三、LLM 技术演进**无法**解决的部分

**意图表达的精确性——这是本质矛盾**

SDD 仍然需要人用领域化的通用语言来描述业务意图，而不是特定技术实现。规范应有清晰结构，用 Given/When/Then 风格定义场景，追求完整而简洁，覆盖关键路径而不是枚举所有情况。

这个要求不会因为模型变强而消失——恰恰相反，随着 AI agent 能够持续自主执行，软件交付的瓶颈已经转移：不再是写代码有多快，而是如何有效表达意图。正如 Adrian Cockcroft 在 QCon SF 上所说，我们正在学习指挥 agent 群体，这是一种从管理个人贡献者到完全不同的组织能力。

**意图的"正确性"无法自动化**

Spec 不只是机械合约，它是人对目的、风险容忍度和取舍的表达。漂移检测能识别系统是否偏离，但无法自行决定这种偏离是可接受的、意外的，还是刻意的——有些漂移是缺陷，有些漂移是演进。在自动化执行和解释性判断的交界处，人的角色再次变得关键。

这句话是整个 SDD 未来走向的核心判断：**执行可以自动化，意义仍然属于人。**

---

## 四、SDD 的三阶段演进路径

基于目前的研究趋势，SDD 的演进路径会分三个阶段展开：

**阶段一（现在 → 2026）：工具化与标准化**

工具景观正在快速碎片化，已经有超过 450 个工具在追踪，各工具在 spec 创建、验证、agent 执行上采取略微不同的方式。 这个阶段的主要任务是收敛——MCP 等标准统一工具集成层，SDD 方法论逐渐从个人技巧变成团队 SOP。

**阶段二（2026 → 2028）：Spec-as-Source 成熟**

SDD 存在三个层次：Spec-First（规范只用一次）、Spec-Anchored（规范是活文档）、Spec-as-Source（规范是主源文件，人只编辑规范，永远不碰代码）。目前所有工具都是 Spec-First，但不是所有都能做到 Spec-Anchored 或 Spec-as-Source。

在这个阶段，会解锁一条路径：框架升级、依赖安全问题等不需要改变规范的维护任务，可以由代码自动重新生成并对照规范验证——这就是"自治性软件维护"。

**阶段三（2028+）：意图层与执行层彻底分离**

SDD 最终形态是把规范变成可执行的、具有强制性的架构元素——架构不再是设计阶段的产物，而成为持续执行的运行时不变量。规范从被动参考材料变成主动控制面，漂移检测成为让系统与意图保持对齐的反馈信号。

这个阶段的关键特征：同一份 spec，可以生成 Java 版本、Go 版本、Python 版本的实现，AI 自主选择最优技术栈——只要保持规范语言和生态系统无关，就可以在生成时提供上下文，指定如何进行生成，从而在 spec 不变的情况下切换到完全不同的技术实现。

---

## 五、最关键的深层判断

代码生成从规范到 LLM 并不是确定性的，这在升级和维护方面带来挑战。规范漂移和幻觉本质上难以避免，仍然需要高度确定性的 CI/CD 实践来保障软件质量。最根本的问题——规范还是代码才是软件开发的最终产物——仍然有待探索。两者会导向完全不同的工作流和开发实践。

这个问题其实是一个更古老问题的新形式：**人类意图和机器执行之间，永远需要一个翻译层。** 过去这个翻译层是编程语言，现在是规范文档，未来可能是更高层的意图描述语言。

该领域正在向"agent 自主管理记忆"演进，区别是程序化记忆（开发者决定存什么）和 agentic 记忆（agent 自己决定）之间的分野。 一旦 agent 能自主决定什么值得记忆、什么可以遗忘，SDD 的文档维护成本会大幅下降——但这同时意味着**人对"记忆了什么"的控制权降低**，这又引入了新的治理问题。

**一句话总结：** SDD 不会被 LLM 解决掉，它会被 LLM 吸收进来——从开发者手动维护的 Markdown 文件，演变为 agent 自动生成、人类只负责 Review 和决策的意图层基础设施。工具消失，方法论留下。
