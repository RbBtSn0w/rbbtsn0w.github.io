## 一、起点：软件工程方法论的本质是什么？

在讨论 SDD 之前，需要先理解一件事：**每一次软件工程方法论的变革，本质上都是对"人与工具之间认知摩擦"的响应。**

瀑布模型诞生于 1970 年代，是对"代码写到一半发现需求错了"这个问题的响应。敏捷宣言诞生于 2001 年，是对"瀑布模型太慢、无法适应变化"的响应。TDD 是对"写完代码才发现设计有问题"的响应。

SDD 诞生的背景，是对一个新问题的响应：**当 AI 能在几分钟内生成几千行代码时，人类工程师的认知能力在哪个环节成为了瓶颈？**

要回答这个问题，必须先理解 AI 编程工具是如何一步步进化的。

---

## 二、AI 编程工具的三个时代

### 第一代（2021-2022）：补全即智能

GitHub Copilot 发布时，它本质上是一个"超级 Tab 键"——你写了函数名，它补全函数体。这个阶段的认知模型很简单：

```
人写意图（函数名/注释） → AI 补全实现 → 人审查几行代码
```

问题不大。人始终在驾驶座，AI 是助手。补全的粒度小，错了也容易发现。

### 第二代（2023）：对话即编程

ChatGPT 和 Claude 的出现改变了交互范式。开发者开始把整个功能需求粘贴给模型，拿回来一段完整的代码。认知模型变成：

```
人描述需求（自然语言） → AI 生成完整代码块 → 人粘贴到编辑器
```

这里出现了**第一个裂缝**：对话是临时的，代码是持久的。每次新开对话，上下文就丢了。开发者开始发现自己在重复解释同一个项目背景。

### 第三代（2024-2025）：Agent 即工程师

Cursor、Claude Code、GitHub Copilot Workspace 等工具出现。AI 不再只是回答问题，而是可以直接修改文件、执行命令、运行测试。认知模型变成：

```
人给出任务描述 → AI 自主规划并执行多步骤操作 → 人验收结果
```

这里出现了**根本性的断裂**：当 AI 在 10 分钟内生成 3000 行代码，横跨 20 个文件，人类如何验收？如何理解？如何维护？

这个断裂，就是 SDD 诞生的真实土壤。

## 三、真正触发 SDD 的三个工程现实

### 现实一：上下文窗口是计算资源，不是记忆

大多数人把 LLM 的上下文窗口理解为"记忆"，这个比喻是错的。更准确的理解是：**上下文窗口是每次推理的工作内存，会话结束就清空，下次推理从零开始。**

这意味着一个基本事实：AI 不记得你上周跟它说的任何事情。

当项目规模增长，代码库积累了数月的决策，一个新的 AI 会话面对这个代码库时，它看到的只是当前的静态快照。它不知道：
- 为什么选了这个架构而不是那个
- 这个函数曾经因为什么 bug 被重写过
- 这个接口设计背后有什么业务约束

这不是 AI 的缺陷，这是所有语言模型的**结构性特征**。

### 现实二：自然语言需求有无法消除的歧义性

"做一个用户登录功能"这句话，一个有 5 年经验的工程师听到，脑子里会自动填充几十个隐含假设：要不要支持第三方登录？密码错误几次要锁定？session 还是 JWT？这些假设来自经验积累，不会被说出口。

但 AI 没有这些经验性假设。它会做一个选择——通常是训练数据中最常见的那个选择——然后不声不响地执行。两天后你发现它实现的方式和你的系统架构完全不兼容，但代码已经写了 2000 行。

**SDD 的 Spec 本质上是把这些隐含假设强制显性化的工具。**

### 现实三：代码速度远超人类理解速度

人类审查代码的速度大约是每小时 100-200 行（认真审查，不是走马观花）。AI 生成代码的速度是每分钟几百行。

当 AI 在一次会话里生成 5000 行代码，人类如果试图逐行审查，需要花几十个小时——这比手写还慢。这就是前面提到的"Review Paralysis"（审查瘫痪）的根源。

**SDD 把审查的重心从"代码是否正确"转移到"代码是否兑现了规范"——这是一个更高效的审查策略，因为审查一份 100 行的 spec 比审查 5000 行代码快得多。**

## 四、为什么不能用 System Prompt 解决？

这是你问题中最核心的部分，值得深入拆解。

System Prompt 能解决的问题，和 SDD 解决的问题，在本质上是不同类型的问题。

### System Prompt 解决的是"角色和风格"问题

System Prompt 是在每次对话开始时注入的固定指令，它能有效地控制：
- AI 的角色定位（"你是一个后端工程师"）
- 输出格式（"用 JSON 回答"）
- 语气风格（"简洁、专业"）
- 行为约束（"不要生成测试代码"）

这些都是**无状态的、对话级别的约束**。

### SDD 解决的是"跨会话知识持久化"问题

看一下你的工程场景：

```
第一次会话：设计了用户权限系统，决定用 RBAC，记录了为什么不用 ABAC
第二次会话（两周后）：修一个 bug，需要理解权限系统的设计意图
第三次会话（一个月后）：新功能要扩展权限系统
```

System Prompt 无法携带第一次会话里做出的决策。你可以在 system prompt 里说"这是一个使用 RBAC 的权限系统"，但你无法把所有历史决策都塞进 system prompt——它有长度限制，而且会变得越来越难维护。

更根本的问题是：**system prompt 是工具配置，不是工程资产。** 它存在于 AI 工具里，不在你的 git 仓库里，不能被团队共享，不能被版本控制，不能在 PR 里被 review。

SDD 的 spec 文件是工程资产，它住在代码库里，和代码一起被版本控制、一起被 review、一起被归档。这是两个完全不同的性质。

### 还有一个更深层的原因：信任链问题

System Prompt 是 AI 工具厂商控制的层。当你在 Cursor 里设置了一段 system prompt，这段 prompt 的格式、语义、生效范围都取决于 Cursor 的实现。如果你换了 Claude Code，这段 system prompt 未必能无缝迁移。

SDD 的 spec 文件是纯文本的、工具无关的。任何 AI 工具只要能读文件，就能使用这份 spec。**这是一种反脆弱的设计**——把关键的工程知识存储在最稳定的介质（纯文本文件）里，而不是绑定在某个工具的私有配置里。

## 五、SDD 真正的思想来源

SDD 并不是凭空出现的，它的思想血统可以追溯到几个不同的方向。

**来自 TDD 的启发：** TDD 的核心洞见是"先写测试，再写代码"——测试描述了期望的行为，代码是让测试通过的实现。SDD 做了一个类似的移位：先写规范，再写代码。规范描述了期望的行为，代码是让规范满足的实现。区别在于，TDD 的"规范"是机器可执行的（单元测试），SDD 的"规范"是 AI 可理解的（自然语言 + 结构化文档）。

**来自 Design by Contract 的继承：** Bertrand Meyer 在 1980 年代提出了"契约式设计"——函数有前置条件、后置条件和不变量，这些构成了函数的"契约"。SDD 的 Acceptance Criteria 本质上是系统级别的契约，只是从代码层提升到了需求层。

**来自 ADR（Architecture Decision Records）的实践：** ADR 是软件工程界已经成熟的实践，记录"为什么做这个决策"。SDD 把 ADR 的精神扩展到了整个开发流程，不只记录架构决策，也记录需求决策、接口决策、实现决策。

**来自 Infrastructure as Code 的类比：** IaC 把基础设施配置从手工操作变成可版本控制的代码文件。SDD 做了一个类似的事情：把开发过程中的"意图"从口头沟通和聊天记录，变成可版本控制的规范文件。

## 六、为什么是"现在"这个时间点？

最后一个值得思考的问题：为什么 SDD 在 2024-2025 年才被广泛讨论，而不是更早？

答案是：以前没有这个必要。

当 AI 只能做代码补全，人始终是主要的生产者，AI 是辅助工具，方法论不需要改变。当 AI 开始能生成完整功能，但生成质量不稳定、错误明显，人工修改是主要工作，方法论还不需要改变。

当 AI 开始能可靠地生成大段可用代码，而且速度远超人类理解速度，**人类在开发过程中的角色从"实现者"变成了"决策者和验收者"**，原有的方法论就失效了——因为原有的方法论假设人是实现代码的主体。

SDD 不是对"AI 时代需要更多文档"这个浅层观察的回应。它是对一个更深层问题的回应：**当实现代码的成本趋近于零，软件工程的价值和瓶颈在哪里？**

答案是：在意图的精确表达、决策的可追溯性、以及跨时间跨人员的知识传递上。而这些，恰好是 SDD 试图用 Spec 文件来解决的问题。

---

**一句话总结：** SDD 的诞生不是因为"AI 编程工具出现了，所以需要新方法论"这么简单。它是因为 AI 把软件开发的速度瓶颈从"代码生产"转移到了"意图表达和知识传递"，而原有的工程方法论对这两个问题几乎没有任何约束和支持——这个真空，是 SDD 存在的根本理由。
